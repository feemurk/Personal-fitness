Context
The current index.html is a 1874-line mobile-only execution app with no data persistence beyond a theme preference. All progress data lives in a session-scoped store object and is lost on page close.
The user wants to:

Replace Notion entirely — the HTML app becomes the one-stop tool for execution, data logging, knowledge, and review
Add Airtable as the persistent backend (2-way sync)
Add a responsive desktop layout (review/knowledge mode)
Redesign the data model to be analytics-ready — structured selectable fields, no open text, domain-specific values informed by all four skill domains (hockey, golf, body-movement, breathwork)
Enable async session saving so sessions can be paused and resumed

Units: lbs / inches. Desktop view: session history table + sparklines (trend lines for pain, confidence, key metrics — no heavy charts, Airtable native reports handle deeper analytics).

Critical Files

/home/user/Personal-fitness/index.html — single file, full rewrite
/home/user/Personal-fitness/CLAUDE.md — coding standards, test gate
/home/user/Personal-fitness/MEMORY.md — user profile and psychological context


Architecture Decisions
Single file
Stay as index.html. No build step, no split files. All CSS + JS inline.
JS Structure
Wrap all JS in one top-level IIFE. Internal module objects:

APP_CONFIG — constants, field option arrays, API URLs
StorageService — localStorage read/write (credentials, draft session, theme)
AirtableService — all API calls (create/patch/fetch for all 3 tables)
SessionMachine — session state machine (IDLE → STARTING → ACTIVE → LOGGING → SAVING → COMPLETE)
UI — namespaced sub-objects: Nav, Today, Log, Progress, Learn, Settings, Snackbar, BottomSheet, ResumeBanner, Setup
App — init() entry point, wires everything together

No onclick= attributes in HTML. All event listeners attached in App.init() via data-* attributes.
CSS Structure
Keep all existing CSS custom properties verbatim. Add:

App shell layout (.app-shell, .content-area)
Bottom nav CSS (mobile, < 768px)
Left rail CSS (desktop, @media (min-width: 768px))
New UI primitives: FAB, Bottom Sheet, Snackbar, Resume Banner, Setup Overlay
Replace .tabs-wrap/.tab-btn/.panel with new nav + panel structure

Security
Airtable PAT + Base ID stored in localStorage only. Entered once via first-launch setup screen. Never in code, never in git. config.example.js is not needed — setup is fully in-app.

Field Description Schema & Info Tooltip
Purpose
Each selectable field in the Log tab carries a machine-readable description — what the field measures, why it matters to this program, and how to pick the right value. This data serves two jobs:

Airtable: populated as the description property on each field via the Metadata API at setup time (visible in Airtable's field configuration)
Frontend: rendered in a tooltip triggered by an info button (ⓘ) on every Log field row

Info Button UI Component
HTML pattern (rendered on each Log field row):

  <div class="log-field-right">
    <span class="log-field-value" id="log-val-Entry_State">—</span>

    <span class="log-field-arrow">›</span>
  </div>
</div>
One shared tooltip element (appended to <body>, positioned dynamically):
html<div id="field-tooltip" class="field-tooltip" role="tooltip" aria-live="polite" hidden>
  <div class="tooltip-header">
    <span class="tooltip-title"></span>
    <button class="tooltip-close" aria-label="Close">✕</button>
  </div>
  <p class="tooltip-what"></p>
  <p class="tooltip-why"></p>
  <ul class="tooltip-options"></ul>
</div>
Behavior (handled in UI.FieldInfo):

mouseenter on ⓘ button → position tooltip near button, populate from FIELD_DESCRIPTIONS, show with data-mode="hover"
mouseleave on ⓘ button → if data-mode="hover" → hide tooltip
click on ⓘ button → if hidden or hover mode → show with data-mode="locked"; if already locked → hide
click on ✕ close button → hide tooltip regardless of mode
Click anywhere outside tooltip while locked → hide tooltip
On mobile (touch devices, window.matchMedia('(hover: none)')) → hover events ignored; click-only toggle

Airtable Schema Creation Integration
When createSchema runs at setup, each field POST to the Metadata API includes a description property populated from FIELD_DESCRIPTIONS[fieldKey].what + ' | ' + FIELD_DESCRIPTIONS[fieldKey].why. This means field purpose is documented inside Airtable itself, visible when building reports.

Session State Machine
IDLE → (start session) → STARTING
STARTING → (POST draft to Airtable) → ACTIVE
ACTIVE → (exercise ticks, phase completion PATCHes) → ACTIVE
ACTIVE → (all exercises done or manual complete) → LOGGING
LOGGING → (user fills Log tab fields) → SAVING
SAVING → (PATCH Complete to Airtable) → COMPLETE
COMPLETE → (reset) → IDLE
localStorage as buffer: Every state transition calls StorageService.setDraftSession() synchronously. Airtable calls are always async try/catch — silent failure, snackbar warning. App never crashes on Airtable errors.
Phase auto-PATCH: Exercise membership per phase is a constant. When all exercises in a phase are ticked, Phases_Completed is PATCHed to Airtable in the background.
javascriptconst PHASE_EXERCISES = {
  'Phase 1 Breathing':   ['ex0', 'ex1'],
  'Phase 2 Scapular':    ['ex2', 'ex3', 'ex4'],
  'Phase 3 Lower Body':  ['ex5', 'ex6', 'ex7', 'ex8'],
  'Phase 4 Power':       ['ex9', 'ex10', 'ex11']
};
Resume banner: On App.init(), check StorageService.getDraftSession() first (fast/sync), then AirtableService.getDraftSessions() (async, catches drafts from other devices).

First-Launch Setup Flow
5-step overlay:

Welcome + purpose explanation
PAT entry (password input, stored to localStorage.AT_PAT)
Base ID entry (stored to localStorage.AT_BASE_ID)
Connection test (GET /meta/bases/{baseId})
Schema creation (POST to Metadata API — creates all 3 tables with all fields and select options)

On subsequent launches: StorageService.isConfigured() → skip setup, go directly to Today tab.
Settings tab has "Re-run setup" button and "Clear credentials" option.

Notes / Constraints

No frameworks. Vanilla HTML/CSS/JS only.
The Today tab and Tonight Mode are protected UX — must remain reachable in one tap.
Log tab uses bottom sheets only — no keyboard input during a sweaty session.
All Airtable errors are silent-with-snackbar — app never crashes offline.
Sport-specific fields (Hockey / Golf) only render in Log when the matching Session_Type is selected.
The left-handed fade in golf is stable. No drills in Learn content should suggest removing it.
